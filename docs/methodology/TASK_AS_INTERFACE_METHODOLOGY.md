# Task as Interface Methodology

## Version

`0.1.3-gllm`

## Core Claim

In StegVerse, the interface is not merely chat.

The task itself is the interface.

A task is a proposed-transition event that can be routed through scoped ecosystem intelligences, converted into candidate evidence, packaged, ingested, reviewed, and either admitted, repaired, denied, quarantined, or failed closed.

## Methodology

```text
task appears
→ task-interface classification
→ scoped entity perspectives identified
→ relevant AI entities produce candidate evidence
→ omissions and conflicts are surfaced
→ evidence is assembled
→ bundle is packaged
→ ingestion receives the bundle
→ destination admissibility decides
→ receipts preserve each link
```

## Entity Role

Scoped AI entities do not vote with authority.

They produce bounded candidate evidence from their jurisdiction:

```text
Core-Lite Entity      execution, receipts, local repo safety
Master-Records Entity reconstruction, provenance, chain continuity
KnowledgeVault Entity continuity, context boundaries, portability
GCAT/BCAT Entity      invariant pressure and admissibility balance
AE Entity             recoverability and boundary coherence
TVC Entity            token, secret, and authority leakage
Site Entity           public claims and exposure risk
Task Interface Entity routing, omission detection, task classification
```

## Required Rule

No ecosystem task may bind until relevant scoped entities have either:

```text
produced candidate evidence
been ruled not applicable with a receipt-bearing reason
failed or timed out with a receipt-bearing reason
or been escalated to human/ecosystem review
```

## No Broad Authority

The task-interface entity is not a ruler, executor, or global administrator.

It may classify, route, assemble, and detect omission.

It may not bind repo state directly and may not bypass ingestion.
